# Tier-2 instruction templates (runner-side)

One Jinja2 template per assertion type that supports tier-2
semantic verification. Each file is named `tier2_<assertion_type>.j2`
where `<assertion_type>` matches the wire field `assertion.type`
(snake_case, lowercase).

## Trust model

- **Outside the boundary (trusted):** the runner's static instruction
  preamble — the framing, the response format, the per-type
  evaluation criterion. Lives in this file's published source code
  and cannot be modified by an attacker who only controls the
  assertion payload or the source code under review.
- **Inside the boundary (untrusted):** values rendered through the
  `| untrusted` Jinja filter. The renderer wraps these in a
  freshly-minted `<BOUNDARY_xxx>...</BOUNDARY_xxx>` tag at the call
  site. The tag is generated by `secrets.token_hex(12)`, used once,
  and discarded — attackers don't know the token and cannot escape
  the boundary.

## Variables

Every template receives the same variable set:

- `ASSERTION_TYPE` — the assertion type string (NOT wrapped in
  `| untrusted` — it's used in the instruction preamble for the
  per-type framing; type is matched against the template filename
  upstream, so an attacker who tampers with this field changes which
  template was selected, not what the LLM sees inside the boundary).
- `ASSERTION_PARAMS` — dict of `params` from the wire payload.
  Always wrapped in `| untrusted` as a JSON-serialized blob.
- `SOURCE_CODE` — text of the source-under-review excerpt. Always
  wrapped in `| untrusted`.

## Response shape

Each template asks the LLM to reply YES / NO / INJECTION_DETECTED on
the first line, then reasoning. `tier2._parse_response` consumes
this shape.
